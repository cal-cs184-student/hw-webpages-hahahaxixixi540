Part 1: Bezier curves with 1D de Casteljau subdivision

Section I: Bezier Curves and Surfaces

A Bezier curve is defined by a set of control points and a parameter \( t \in [0,1] \). To evaluate a point on the curve, we use de Casteljau’s algorithm, which repeatedly performs linear interpolation between adjacent control points.

\[ p_i' = (1 - t)p_i + t p_{i+1} \]

Each subdivision step reduces the number of points by one. Repeating this process generates intermediate subdivision levels until only a single point remains. That final point lies exactly on the Bezier curve at parameter \( t \).

Implementation

I implemented BezierCurve::evaluateStep(...) to perform exactly one subdivision level. For each adjacent pair of points, the function computes the linear interpolation using the member variable t and stores the results in a new vector. The output vector therefore has size points.size() - 1. If fewer than two points are provided, the function returns an empty vector.

Part 2: Bezier Surfaces with Separable 1D de Casteljau

How de Casteljau Extends to Bezier Surfaces

A Bezier surface is defined by an \( n \times n \) grid of control points \( P_{ij} \) and two parameters \( u, v \in [0,1] \).

The separable 1D de Casteljau algorithm evaluates the surface in two stages.

Row Evaluation (parameter \( u \))

Each row of \( n \) control points

\[ P_{i0}, P_{i1}, \dots, P_{i(n-1)} \]

defines a Bezier curve parameterized by \( u \). Using the same recursive interpolation from Part 1,

\[ p_i' = (1-u)p_i + u p_{i+1}, \]

we evaluate each row to produce a single point \( P_i \).

Column Evaluation (parameter \( v \))

The resulting points

\[ P_0, P_1, \dots, P_{n-1} \]

form another Bezier curve, now parameterized by \( v \). Applying the same 1D de Casteljau algorithm again produces a final point

\[ P = B(u,v), \]

which lies on the Bezier surface. Thus, a 2D Bezier surface evaluation reduces to two applications of the 1D de Casteljau algorithm.

Implementation Details

1️⃣ BezierPatch::evaluateStep(...)

This function performs one subdivision level in 3D. It mirrors the 2D version from Part 1 but operates on Vector3D. For each adjacent pair of points, it computes:

\[ p_i' = (1-t)p_i + t p_{i+1} \]

The function returns a vector of size points.size() - 1.

2️⃣ BezierPatch::evaluate1D(...)

This function fully evaluates a Bezier curve in 3D. It repeatedly calls evaluateStep(...), and each iteration reduces the number of points by one. When only one point remains, that point is returned as the final point on the Bezier curve at parameter \( t \).

3️⃣ BezierPatch::evaluate(u, v)

This function evaluates the full Bezier surface:

  • First, for each row controlPoints[i], I call evaluate1D(..., u), which produces one point per row.
  • These points form a temporary vector pointsAlongV.
  • I then call evaluate1D(pointsAlongV, v), which produces the final surface point at \( (u,v) \).

This exactly implements the separable 1D de Casteljau algorithm described in lecture.

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-weighted vertex normals

In Part 3, I implemented area-weighted vertex normals so the mesh can use Phong shading instead of flat shading. The main idea is: for a given vertex, look at all triangles that touch it, add up their face normals weighted by area, and then normalize the final sum.

Implementation idea

Because the mesh uses a half-edge structure, each vertex stores one outgoing halfedge. Starting from that halfedge, I walk around the vertex’s one-ring using h = h->twin()->next() until I come back to the start. For each halfedge on this loop, I get its face and skip it if it is a boundary face.

For a valid triangle face, I read the three vertex positions from the local halfedge cycle and compute the (unnormalized) face normal using a cross product:

\[ \mathbf{n}_f = (\mathbf{p}_1 - \mathbf{p}_0) \times (\mathbf{p}_2 - \mathbf{p}_0) \]

This vector has magnitude proportional to the triangle area (it is basically \(2 \times \text{area}\)), so adding these cross products automatically gives an area-weighted average. Finally, I normalize the accumulated vector to return a unit vertex normal. If the accumulated vector is zero (degenerate case), I just return a zero vector.

Results

I tested on dae/teapot.dae. With default shading, the surface looks faceted. After pressing Q to toggle area-averaged normals (which calls my Vertex::normal()), the teapot becomes noticeably smoother.

Part 4: Edge flip

In Part 4, I implemented HalfedgeMesh::flipEdge(EdgeIter e0). The goal is to flip an interior edge shared by two triangles, changing the diagonal from (b, c) to (a, d), while keeping the same number of mesh elements (no allocation or deletion).

My code follows this local 2-triangle pattern. I start with h0 = e0->halfedge() and h1 = h0->twin(). I immediately return if the edge or either adjacent face is on the boundary. Then I collect the six halfedges: h2 = h0->next(), h3 = h2->next() (in face f0), and h4 = h1->next(), h5 = h4->next() (in face f1). Using these, the four vertices are read as vb = h0->vertex(), vc = h1->vertex() (old edge endpoints), va = h3->vertex() (opposite in f0), and vd = h5->vertex() (opposite in f1).

After that, I rewire pointers so h0/h1 now represent the new diagonal (a, d) by setting their vertex pointers to va and vd. I rebuild the two 3-cycles using setNeighbors(...): new f0 loop is h0 -> h5 -> h2, and new f1 loop is h1 -> h3 -> h4. Finally I update the representative halfedge() pointers for e0, both faces, the four surrounding edges, and the four vertices so later traversals start from valid halfedges.

Debug note

At first, one flip looked fine, but flipping the same edge back sometimes broke the mesh. I realized I rewired the 6 halfedges but forgot to refresh some representative pointers (like face->halfedge() / vertex->halfedge()). After I explicitly updated those pointers (plus e0->halfedge()), flip-back became stable.

Implementation

Starting from h0 = e0->halfedge() and h1 = h0->twin(), I read the two endpoints (b, c) and the two opposite vertices (a, d). I then insert a new vertex m at the midpoint of the original edge:

Part 6: Loop subdivision for mesh upsampling

Loop Subdivision Implementation

Overview

Loop subdivision refines a triangle mesh using a 4-1 subdivision scheme followed by weighted vertex updates. Each iteration consists of three phases:

  1. Precompute new vertex positions without modifying connectivity.
  2. Split edges and flip necessary edges.
  3. Update vertex positions.

Separating geometry computation from connectivity changes ensures correctness and prevents mixing original and updated values.

Interior Vertex Update Rule

For an interior vertex of degree \( n \):

\[ v' = (1 - n u)v + u \sum_{i=1}^{n} v_i \]

where

\[ u = \begin{cases} \frac{3}{16}, & n = 3 \\ \frac{3}{8n}, & \text{otherwise} \end{cases} \]

This weighted averaging smooths the mesh and guarantees smooth limit behavior.

Edge Vertex Update Rule

For an interior edge shared by triangles (A, C, B) and (A, B, D):

\[ v_{new} = \frac{3}{8}(A + B) + \frac{1}{8}(C + D) \]

For boundary edges:

\[ v_{new} = \frac{1}{2}(A + B) \]

4-1 Subdivision Procedure

  • Store all original edges.
  • Split every original edge.
  • Mark new vertices and edges.
  • Flip new edges connecting one old and one new vertex.
  • Do not flip edges created directly from splits.

After connectivity refinement, all vertex positions are updated using the precomputed values.

Behavioral Observations

Smoothing Effect

Repeated subdivision rounds sharp corners and converges toward a smooth limit surface. Loop subdivision is an approximating scheme, meaning original vertices move.

Cube Subdivision Behavior

After repeatedly subdividing cube.dae:

  • The cube becomes rounded.
  • Faces curve inward.
  • Slight asymmetry appears.

This asymmetry arises because square faces are triangulated with diagonals. Since Loop subdivision depends on connectivity, diagonal orientation affects smoothing.

Preprocessing for Symmetry

By flipping diagonals to enforce symmetric triangulation, the resulting subdivision becomes more uniform and visually symmetric.

Extra Credit: Boundary Handling

For boundary vertices, I implemented the special boundary rule:

\[ v' = \frac{3}{4}v + \frac{1}{8}(v_0 + v_1) \]

Without this rule, boundary vertices shrink inward. With the correct boundary stencil, the boundary remains smooth and stable.
